## Especificación Técnica: Patrón Estrategia para Multitenancy en Trust Score Engine

### 1. Objetivo

Implementar el patrón **Strategy** para desacoplar la lógica de cálculo del Trust Score según la configuración de cada **Tenant**. Esto permitirá que coexistan dos motores:

1. **Legacy Delta Engine:** Cálculo actual basado en diferencias simples.
2. **Advanced Trajectory Engine:** Cálculo basado en regresión lineal y snapshots históricos.

### 2. Estructura de Clases (DDD)

#### A. Definición de la Abstracción

En `RentGuard.Core.Business/Modules/TrustScore/Domain/`:

```csharp
public interface ITrustScoreCalculationStrategy
{
    // Identificador único de la estrategia
    string StrategyName { get; }
    
    // Método principal de ejecución set-based
    Task CalculateInsightsAsync(Guid tenantId, CancellationToken ct);
}

```

#### B. Implementaciones de Estrategia

* **`LegacyDeltaStrategy`**: Implementará la lógica actual. No requiere de la tabla `TrustScoreSnapshots` para el cálculo profundo, solo comparativas inmediatas.
* **`AdvancedTrajectoryStrategy`**: Orquestará la ejecución del script SQL `UpdateTrustInsights.sql`. Utilizará la infraestructura de tablas temporales y procesamiento por lotes.

### 3. Resolución por Tenancy (Infraestructura)

Se debe implementar un **Resolver** que actúe como una factoría dinámica basada en los metadatos del Tenant.

```csharp
public class TrustScoreEngineResolver : ITrustScoreEngineResolver
{
    private readonly IEnumerable<ITrustScoreCalculationStrategy> _strategies;
    private readonly ITenantSettingsRepository _settingsRepository;

    public async Task<ITrustScoreCalculationStrategy> GetStrategyForTenantAsync(Guid tenantId)
    {
        var engineType = await _settingsRepository.GetTrustEngineTypeAsync(tenantId);
        
        return _strategies.FirstOrDefault(s => s.StrategyName == engineType) 
               ?? _strategies.First(s => s.StrategyName == "Legacy");
    }
}

```

### 4. Integración con el Plan de Implementación SQL

El script `UpdateTrustInsights.sql` debe diseñarse para ser **Tenant-Aware** y ejecutarse bajo demanda de la estrategia avanzada:

* **Atomicidad:** La estrategia debe abrir una transacción (`TransactionScope` o `DbTransaction`).
* **Set-based:** La estrategia avanzada debe cargar los IDs de los usuarios afectados en una tabla temporal (`#UsersToProcess`) antes de invocar la lógica de regresión lineal para maximizar la eficiencia del motor SQL.

### 5. Configuración de Metadatos (Base de Datos)

Modificar la tabla de configuración de Tenants o crear una nueva:

| Columna | Tipo | Descripción |
| --- | --- | --- |
| `TrustEngineType` | `VARCHAR(50)` | Valores: 'Legacy' o 'AdvancedTrajectory'. |
| `HistoricalWindow` | `INT` | Meses a considerar para la regresión (solo para Advanced). |

### 6. Criterios de Aceptación (TDD)

1. **Aislamiento:** Las pruebas unitarias deben confirmar que invocar el cálculo para un Tenant con estrategia `Legacy` no genera registros en la tabla `TrustScoreSnapshots` si no es necesario.
2. **Polimorfismo:** El `TrustScoreService` no debe contener sentencias `if/else` para decidir la lógica de cálculo; debe delegar totalmente al `ITrustScoreCalculationStrategy` resuelto.
3. **Rendimiento:** El motor `Advanced` debe ejecutar sus cambios en lotes DDL/DML, procesando todos los usuarios del Tenant en una sola operación de base de datos.

### 7. Instrucciones para la IA (Contexto de Desarrollo)

> [!IMPORTANT]
> Al generar código, asegúrate de que todas las implementaciones de `ITrustScoreCalculationStrategy` sigan el principio de **Single Responsibility**. La lógica de persistencia pesada debe residir en el repositorio, mientras que la estrategia decide *qué* procedimiento invocar según el contexto del Tenant.

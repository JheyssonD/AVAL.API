# 📘 Consistency & Temporal Policy — RentGuard AI

Esta especificación define las garantías de consistencia de los datos y las reglas para el manejo del tiempo y el orden de los eventos en un entorno distribuido.

---

## 1. Matriz de Consistencia

| Operación | Tipo de Consistencia | Justificación |
| :--- | :--- | :--- |
| **Aprobación de Pago** | **Fuerte (Transaccional)** | Evita discrepancias en el ledger financiero. |
| **Actualización TrustScore** | **Eventual (Segundos)** | Desacoplada del flujo de pago para escalabilidad. |
| **Dashboard / Analytics** | **Eventual (Segundos/Minutos)** | Optimizada para lectura masiva. |
| **Notificaciones** | **Eventual (Segundos)** | Naturaleza de red (WhatsApp/Meta). |

---

## 2. Política Temporal y Anti-Clock Drift

### 2.1 Autoridad de Timestamps
- **Regla T-01 (Server Authority)**: La única fuente de verdad para la marca de tiempo de persistencia es el servidor de base de datos SQL Server (`GETUTCDATE()`).
- **Regla T-02 (UTC Only)**: Prohibido el uso de zonas horarias locales en la capa de negocio o persistencia.

### 2.3 Clock Skew Tolerance
- **Regla T-03**: Se permite una discrepancia de hasta **120 segundos** entre sistemas distribuidos (ej. Webhook de WhatsApp vs Servidor RentGuard). Mensajes fuera de este rango requieren auditoría manual.

---

## 3. Manejo de Orden (Event Ordering)

### 3.1 Fuera de Secuencia
- **Regla O-01**: Toda entidad posee una marca de tiempo de última actualización (`LastUpdatedUtc`).
- **Regla O-02**: Si llega un evento con `occurredOnUtc` menor a la última actualización, se descarta el cambio de estado pero se registra en el log de auditoría (Idempotencia).

---

## 4. Garantías de Consistencia Atómica (Financiera)
- La transacción que aprueba un pago **DEBE** incluir:
  1. Cambio de estado del `Payment`.
  2. Inserción del registro en el `Ledger`.
  3. Inserción del mensaje en el `Outbox`.
  4. Actualización del `CreditBalance`.
  *Si falla uno, fallan todos.*

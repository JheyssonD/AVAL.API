# 📘 100. Features & Processes Documentation — RentGuard AI

Este documento describe las funcionalidades principales y los procesos de negocio del sistema RentGuard AI.

## 1. Captura de Pagos vía WhatsApp
El sistema facilita el reporte de pagos para los inquilinos a través de una integración fluida con WhatsApp.

### Flujo del Proceso
1. **Recepción de Mensaje**: Un webhook recibe un mensaje del inquilino que contiene un comprobante de pago (imagen).
2. **Extracción de Metadatos**: El sistema extrae el `MessageId` y el `TenantId` (número de teléfono).
3. **Almacenamiento**: El comprobante se almacena y se crea un registro en la tabla `Payments` con `Status = Received`.

## 2. Pipeline de Validación con IA
Validación automatizada de comprobantes de pago utilizando Vision AI.

### Pasos
1. **Procesamiento OCR**: Vision AI extrae datos clave: Monto, Fecha, Referencia.
2. **Verificación de Confianza**: Si la confianza de la IA es inferior al 85%, el pago se marca para `PendingReview` (Revisión Pendiente).
3. **Análisis de Discrepancia**: El monto extraído se compara con el monto esperado del contrato. Si la diferencia es > 0.5%, pasa a `PendingReview`.
4. **Detección de Duplicados**: El sistema genera un hash del comprobante para evitar envíos duplicados.

## 3. Motor de TrustScore (Arquitectura Multitenant)
El sistema implementa un **Strategy Pattern** que permite a cada Landlord (Tenant) elegir el motor de cálculo que mejor se adapte a sus necesidades.

### Estrategias Disponibles:
1. **Legacy Delta Engine**: Basado en sumas y restas simples de puntos ante eventos de pago.
2. **Advanced Trajectory Engine**: Utiliza **Regresión Lineal Simple** sobre snapshots históricos para detectar tendencias de comportamiento, estabilidad y madurez del inquilino.

### Motor Avanzado (Trajectory)
- **Vector de Tendencia**: Calculado mediante la pendiente de la recta de regresión de los últimos puntajes.
- **Volatility Flag**: Se activa ante caídas drásticas (>200 pts) y requiere un periodo de recuperación de 3 meses.
- **Eventos**:
    - **Pago Anticipado**: +15 puntos.
    - **Pago Puntual**: +10 puntos.
    - **Pago Tardío**: -20 puntos.
    - **Pago Parcial**: 0 puntos (Tratado como incompleto).
    - **Pago Rechazado**: -5 puntos.
- **Procesamiento por Lotes**: La lógica de cálculo se ejecuta de forma **Set-based** directamente en SQL para máxima eficiencia.

## 4. Flujo de Revisión Manual
Maneja casos excepcionales donde la IA no puede validar de forma autónoma un pago.

### Condiciones de Activación
- Baja confianza de la IA.
- Discrepancia en el monto.
- Artefactos visuales en el comprobante.
- Marcado por el administrador.

### Acciones
- **Aprobar**: Confirma manualmente el pago, actualiza el estado a `Approved` y activa la actualización del TrustScore.
- **Rechazar**: Marca el pago como `Rejected` y notifica al inquilino.

## 5. Gestión de Propiedades y Contratos
Funcionalidad central para la organización de los activos inmobiliarios.

### Capacidades
- **Registro de Propiedades**: Almacenamiento de dirección, renta mensual, moneda y dueño (LandlordId).
- **Control de Arrendamiento**: Vinculación de inquilinos con propiedades, definiendo fechas de inicio/fin y el día de pago del mes.
- **Validación de Reglas**: El sistema asegura que los días de pago sean válidos (1-31) y que los montos de renta sean positivos.

## 6. Patrón SQL Outbox (Arquitectura) [IMPLEMENTED]
Garantiza la entrega confiable de mensajes y la consistencia eventual entre módulos.

### Implementación
- **Tabla OutboxMessages**: Almacena eventos para ser procesados.
- **Worker de Segundo Plano**: `OutboxProcessor` consulta la tabla cada 5 segundos y ejecuta los handlers correspondientes (ej. `PaymentApprovedEventHandler`).
- **Desacoplamiento**: El flujo de pagos ya no actualiza el TrustScore directamente; lo hace a través de eventos en el Outbox.

## 7. Persistencia y Aislamiento (Multi-tenancy) [IMPLEMENTADO]
El sistema aplica un aislamiento estricto de datos por Landlord/Tenant mediante **Entity Framework Core**.

### Mecanismos de Aislamiento
- **Global Query Filters**: El `RentGuardDbContext` aplica automáticamente un filtro `WHERE TenantId = @TenantId` a todas las entidades (`Properties`, `Leases`, `Payments`, `TrustScoreHistory`, `TrustScoreSnapshots`). Esto garantiza que un Tenant nunca pueda ver ni modificar datos de otro, incluso si el desarrollador olvida filtrar manualmente.
- **TenantMiddleware**: Resuelve el `TenantId` desde el encabezado HTTP `X-Tenant-Id` y lo propaga mediante el `ITenantContext`.

### Integridad de Datos (Concurrencia)
- **Optimistic Concurrency**: Se utiliza una propiedad `RowVersion` (tipo `byte[]` con `timestamp/rowversion` en SQL Server) en entidades críticas como `Payment` y `Lease`. 
- **Prevención de Overwrites**: Si dos administradores intentan actualizar el mismo registro simultáneamente, el sistema detecta el conflicto mediante EF Core y lanza una `DbUpdateConcurrencyException`, protegiendo la integridad financiera de los pagos.

### Estrategia de Acceso a Datos
- **EF Core (Predeterminado)**: Para operaciones CRUD, lógica de negocio y cumplimiento de multi-tenancy.
- **Dapper (Optimización)**: Reservado exclusivamente para consultas de alto rendimiento o scripts complejos de procesamiento por lotes (ej. Trajectory Engine masivo) donde el overhead de EF sea crítico.

# 📘 Business Rules Specification — RentGuard AI (Enterprise Gold Master)

Este documento define el catálogo maestro de reglas de negocio (BRD). Estas reglas son leyes absolutas que rigen el comportamiento del sistema, garantizando la integridad financiera, la transparencia y el aislamiento multi-tenant.

---

## 1. TrustScore (Sistema Reputacional)

### 1.1 Límites y Escalas
- **Rango**: [0, 1000]. Inicial: 80.
- **Suelo de Riesgo**: Ningún evento baja el score de 0.

### 1.2 Reglas de Puntuación
- **BR-TS-01 (Anticipo)**: >48h antes: **+15**.
- **BR-TS-02 (Puntual)**: Hasta el vencimiento: **+10**.
- **BR-TS-03 (Gracia)**: Días 1-5 post-vencimiento: **+2**.
- **BR-TS-04 (Tardío)**: Días 6-15 post-vencimiento: **-20**.
- **BR-TS-05 (Muy Tardío)**: >15 días: **-50**.
- **BR-TS-06 (Fraude)**: Confirmado: **Score = 0 + Bloqueo**.

### 1.3 Política de Acumulación Parcial
- **BR-TS-07 (Bonificación por Saldo)**: Solo se bonifica cuando el `TotalPaid` del periodo sea `>= MonthlyRent`.
- **BR-TS-08 (Fecha de Evaluación)**: Para efectos de puntos, se toma la **timestamp del último pago necesario** para completar la renta del mes.

### 1.4 Compensación Reputacional (Audit Adjustments)
- **BR-TS-09 (Reversión Administrativa)**: Si un administrador revierte un pago que generó puntos, el sistema debe disparar un evento de compensación que reste exactamente los puntos otorgados originalmente.
- **BR-TS-10 (Snapshots STR-02)**: Generación el **último día de cada mes a las 23:59:59 UTC**. El snapshot captura el score consolidado y es **Inmutable**. Los snapshots históricos jamás pueden recalcularse retroactivamente para proteger la integridad de los analytics.
- **BR-TS-11 (Compensation Lineage)**: Todo ajuste compensatorio debe mantener un **Lineage** (vínculo) con el ID del ajuste o pago padre, permitiendo rastrear cadenas de correcciones administrativas complejas.

## 2. Gestión de Pagos y Liquidación (Settlement)

### 2.1 Ciclo de Vida del Pago (Lifecycle)
| Estado | Descripción | Terminal |
| :--- | :--- | :--- |
| **Received** | Webhook de WhatsApp recibido exitosamente. | No |
| **Processing** | OCR y validación de IA en ejecución. | No |
| **PendingReview** | Requiere intervención manual (Bajo confidence/Fraude). | No |
| **Approved** | Validado y aplicado al balance del contrato. | **Sí** |
| **Rejected** | Invalidado (Imagen ilegible/Datos incorrectos). | **Sí** |
| **Duplicate** | Hash de imagen detectado. | **Sí** |
| **Failed** | Error técnico irreversible (ej. Payload corrupto). | **Sí** |

### 2.2 Gobernanza del Estado "Approved"
- **BR-PY-01 (Naturaleza)**: El estado `Approved` es **Workflow Terminal** (no puede volver a estados previos) y **Ledger Immutable** (el registro original no se edita). Es **Compensable** únicamente mediante nuevos registros de ajuste.

### 2.3 Política de Re-envío (Rejected Resubmission)
- **BR-PY-02 (New Entity)**: Un pago `Rejected` no se "re-abre". El usuario debe generar un **nuevo intento de pago** (nueva entidad `Payment`) con su propia correlación y hash.

### 2.4 Correlación de Comprobantes (Voucher Policy)
| Escenario | Resultado |
| :--- | :--- |
| Mismo Hash (SHA-256) | `Duplicate` (Terminal) |
| Diferente Hash + Misma Ref. Bancaria | `PendingReview` |
| Mismo Monto + Mismo Timestamp Banco | `FraudSuspected = true` |

### 2.5 Invariante de Consistencia Atómica
- **BR-PY-03 (All-or-Nothing)**: La persistencia del `Payment`, la actualización del `CreditBalance` y el cálculo del `TrustScore` deben ocurrir dentro de una **única transacción lógica atómica**.

### 2.2 Prioridad de Validación
En caso de múltiples hallazgos, el resultado se determina por:
1. **Duplicate**: Terminal inmediato (Gana siempre).
2. **FraudFlag**: `PendingReview` con alerta de seguridad.
3. **LowConfidence**: `PendingReview` estándar.

### 2.3 Reglas de Fallo Técnico (Retryable vs Failed)
| Error | Estado Resultante | Acción |
| :--- | :--- | :--- |
| Timeout OCR / IA | `Processing` (Retry) | Reintento exponencial (5 veces). |
| Blob Unavailable | `Processing` (Retry) | Reintento exponencial (5 veces). |
| SQL Deadlock | `Processing` (Retry) | Reintento inmediato. |
| Payload Corrupto | `Failed` | No reintento. Notificar Admin. |

### 2.4 Gobernanza de PendingReview
- **BR-PY-04 (SLA)**: Los pagos en `PendingReview` deben ser procesados en **< 24 horas**.
- **BR-PY-05 (Propietario)**: Solo el Landlord del Tenant correspondiente puede aprobar/rechazar.
- **BR-PY-06 (Auto-Reject)**: Si un pago permanece en `PendingReview` > 72h, el sistema lo marca automáticamente como `Rejected`.

## 3. Gobernanza Financiera (Credit Balance)

### 3.1 Propiedad y Deuda
- **BR-CB-01 (Ownership)**: El `CreditBalance` pertenece al **Lease (Contrato)**.
- **BR-CB-02 (Deuda Acumulada)**: El MVP **NO soporta** intereses compuestos ni recargos por mora automáticos. La deuda se limita estrictamente a la sumatoria de rentas mensuales no cubiertas.

---

## 4. IA y Antifraude (Fraud Rules)

### 4.1 Fraude como Metadata
- **BR-IA-01 (Flagging)**: El fraude no es un estado, sino un **Flag Técnico** (`FraudSuspected = true`). 
- **BR-IA-02 (Colisión)**: El hash duplicado se evalúa en el ámbito: `TenantId + SHA256 + 90 días`.

---

## 5. Auditoría Administrativa (Override Rules)

### 5.1 Política de Inmutabilidad (Append-only Ledger)
- **BR-AD-01 (No Modification)**: NUNCA se modifica un pago en estado `Approved`.
- **BR-AD-02 (Evento Compensatorio)**: Todo ajuste es un nuevo registro con referencia al original.

### 5.2 Idempotencia y Scope Administrativo
- **BR-AD-03 (Idempotencia)**: Todo `Adjustment` debe referenciar un `OriginalPaymentId` y un `AdjustmentType`. No se permite más de un ajuste del mismo tipo para el mismo pago (evita doble reversión).
- **BR-AD-04 (Landlord Scope)**: Acceso restringido exclusivamente a sus Tenants vinculados.
- **BR-AD-05 (Global Admin)**: Permiso de intervención cross-tenant únicamente para soporte técnico y resolución de disputas legales.
- **BR-AD-06 (Correlation Governance)**: Todo flujo distribuido debe propagar obligatoriamente:
  - `CorrelationId`: Flujo completo de negocio.
  - `CausationId`: Evento o mensaje origen.
  - `TraceId`: Trazabilidad técnica de observabilidad.

### 5.3 Auditoría Legal (Audit Log)
- **BR-AD-06 (Mandatory Logging)**: Toda acción manual (Aprobación, Rechazo, Ajuste) debe registrar:
  - **Actor**: ID del usuario responsable.
  - **Motivo**: Justificación textual obligatoria.
  - **Timestamp**: Fecha/hora UTC.
  - **State Snapshot**: Estado anterior y posterior (Before/After).

---

## 6. Integridad de Datos y Contratos

### 6.1 Política de Timestamps
- **BR-ID-01 (UTC Only)**: Persistencia y lógica de negocio siempre en **UTC**.

### 6.2 Transición de Contratos (Overlap)
- **BR-EX-04 (Same-Day Transition)**: Si un contrato termina el mismo día que inicia otro en la misma propiedad, el contrato saliente (`Expired`) tiene prioridad de ocupación hasta las 23:59:59 UTC. El nuevo contrato se activa legalmente a las 00:00:00 UTC del día siguiente.

---

## 7. Políticas de Persistencia (Deletion Strategy)

| Entidad | Estrategia | Justificación |
| :--- | :--- | :--- |
| **Payments** | **Inmutable** | Integridad financiera y auditoría. No eliminación. |
| **Lease** | **Soft Delete** | Recuperación ante errores de carga. Excluido de vistas `Active`. Restore permitido. No cascade. |
| **Tenant** | **Archived** | Conservación de datos históricos por 5 años. Blobs x5 años. Leases terminados. Balances congelados. |
| **Properties** | **Archived** | Mantenimiento de trazabilidad de activos. |

### 7.2 Isolation & Restoration Rules
- **BR-PD-05 (Isolation Boundary)**: Toda consulta a la base de datos **DEBE** incluir el `TenantId` de forma obligatoria (automatizado vía Global Query Filters). Los joins cross-tenant están prohibidos excepto en procesos de auditoría global debidamente autorizados.
- **BR-PD-06 (Archival Restoration)**: Un Tenant en estado `Archived` puede ser restaurado a `Active` únicamente mediante un proceso de validación legal y técnica manual por el **Global Admin**. No existe auto-restauración.

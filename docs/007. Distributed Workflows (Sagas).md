# 📘 Distributed Workflow Specification (Sagas) — RentGuard AI

Esta especificación define cómo el sistema maneja procesos de negocio de larga duración o que involucran múltiples servicios, garantizando la consistencia mediante compensaciones.

---

## 1. Patrón: Coreografía de Eventos

RentGuard AI utiliza **Coreografía** basada en el Outbox para orquestar sus flujos. No existe un orquestador central; cada servicio sabe cómo reaccionar a los eventos de dominio.

---

## 2. Flujo Crítico: Aprobación de Pago y TrustScore

### Paso 1: Aprobación (Módulo Payments)
- **Acción**: `ApprovePaymentCommand`.
- **Resultado**: `PaymentApprovedEvent` publicado en Outbox.
- **Fallo**: Transacción atómica falla. No se publica evento.

### Paso 2: Cálculo de Score (Módulo Reputation)
- **Trigger**: Suscripción a `PaymentApprovedEvent`.
- **Acción**: Ejecutar estrategia de TrustScore.
- **Side Effect**: `TrustScoreUpdatedEvent`.
- **Fallo**: Reintento automático vía Outbox (hasta 5 veces). Si persiste, mover a `Quarantined`.

---

## 3. Estrategias de Compensación

| Fallo en Paso | Acción Compensatoria | Responsable |
| :--- | :--- | :--- |
| **Error en TrustScore** | Registrar inconsistencia en Auditoría. El Admin puede forzar recálculo manual. | Reputación Service |
| **Detección de Error Post-Aprobación** | Generar un **Adjustement Event** negativo para revertir el balance y los puntos. | Ledger / Admin |

---

## 4. Retries y Resiliencia
- **Exponential Backoff**: 2s, 4s, 8s, 16s, 32s.
- **Manual Intervention**: Mensajes en `Quarantined` requieren acción del Global Admin para ser re-procesados tras corregir la causa raíz.
